Behavior Driven Development

using RSpec

The Prereq: Test Driven Development

Test driven development (TDD) is a software development process that relies on writing automated tests first, and then following up with writing the implementation. This process is often referred to as *Red-Green-Refactor*:

Shortcomings of TDD

Dan North, the pioneer of BDD, had this to say when using TDD:

I had a problem. While using and teaching agile practices like test-driven development (TDD) on projects in different environments, I kept coming across the same confusion and misunderstandings. Programmers wanted to know where to start, what to test and what not to test, how much to test in one go, what to call their tests, and how to understand why a test fails.

What is Behavior Driven Development?

Behavior-driven development (or BDD) is an agile software development technique devised as a response to the issues he encountered with TDD:
  • Where to start
  • What to test and what not to test
  • How much to test in one go
  • What to call the tests
  • How to understand why a test fails

1. Test method names should be sentences


  • Test names are whole sentences that include an `actor` and start with the word `should`.
  • Understandable by non-developers, leveraging the ubiquitous language of the domain (everyone uses the same nomenclature).

2. Emphasize behavior over testing


  • Solves the difficult question, "What to test?"
  • Defines an expected set a behavior rather than implementation specific fuctionality..

3. Every story should have a user facing acceptance criteria. `Outside-in` testing.


  • Stakeholders should be able to accept the story using some interface.
  • Allows for quick feedback from stakeholders on whether or not the assumptions about the behavior are correct.
  • Encourages YAGNI

The Flow of BDD

To start ouside in, one writes a request spec emulating an actor.

Request Spec Level

  1. Write the request spec
  2. Run said request spec, watch it go red
  3. Write view implementation of request spec
  4. Re-run said request spec, watch it still go red
  5. Dive outside-into controllers

Controller Spec Level

  1. Write the controller spec
  2. Run controller spec, watch it go red
  3. Write implementation of controller
  4. Re-run said controller spec, watch it still go red
  5. Dive outside-into models

Model Spec Level

  1. Write the model spec
  2. Run model spec, watch it go red
  3. Write implementation of model
  4. Re-run said model spec, watch it go green
  5. Completed inner most implementation, begin to pop back outside

Popping back up

  1. Run the controller spec, refactor code until green.
  2. Run the request spec, refactor code until green.
  3. You're done!

Summary

  1. You now have a user facing, acceptable story, with tests all the way down the stack.
  2. Only implemented what's needed to get green (YAGNI).
  3. Clear demonstration of outside - in

BDD in Action

To facilitate BDD, RSpec builds a DSL to support the BDD nomenclature language: 


        describe "after user submits a post" do
          it "should create a new post" do
          ...
          end
        end

BDD in Action

To facilitate BDD request specs, capybara aliases [describe] and [it] to [feature] and [scenario]. 


        feature "user posts" do
          scenario "user creates a post" do
          ...
          end
        end
Scenarios are often larger in scope than simple it { } blocks.
